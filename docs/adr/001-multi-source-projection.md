# ADR-001: 複数データソースからの投影 (Projection) パターン

## ステータス

Accepted

## コンテキスト

Claude Code セッションの状態は単一のソースからは得られない:

- **JSONL** は対話履歴・トークン使用量を持つが、リアルタイム更新ではない (ファイル書き込みベース)
- **Hook イベント** は Status 遷移を即座に通知するが、トークンやプロンプトは持たない
- **PTY 出力** はリアルタイムだが構造化されていない (ターミナルエスケープシーケンスの生バイト)
- **Store** は永続化したメタデータだが、起動間の復元用でしかない

当初は Session に対する setter を場当たり的に呼んでいたため、「どのフィールドをどのソースが所有するか」が不明瞭だった。

## 決定

**DataSource 型で各ソースを明示し、Apply* メソッドで投影ルールをカプセル化する。**

- `DataSource` 列挙型 (Store, JSONL, Hook, PTY) でソースを分類
- 各ソースは Session の特定フィールドのみを更新する (所有権)
- `ApplyJSONLTokens()`, `ApplyFileActivity()`, `ApplyBookmark()` 等の専用メソッドが更新を適用
- TUI は `Snapshot()` を通じてのみ Session を読む (ロックフリー)

## 結果

**良い点**:
- フィールドの所有権が明確 (「Prompt を変えるのは JSONL だけ」)
- 投影ルールの変更が Apply* メソッドに局所化される
- 新しいソースの追加が容易 (新しい Apply* メソッドを追加するだけ)

**悪い点**:
- 4つのソースが非同期で更新するため、一時的な不整合がありうる (例: Hook で Status=Completed だがトークン集計はまだ)
- デバッグ時に「この値はどこから来たか」の追跡にログが必要
